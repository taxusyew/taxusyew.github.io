Chromium多进程框架

Multi-process Architecture

这篇文档描述了 Chromium 的高层次架构。

问题

建造一个从不崩溃或宕机的渲染引擎几乎是不可能的。同样建造一个有着完美安全性的渲染引擎也几乎是不可能的。

从某些方式上来看，当前浏览器的状况和从协作单用户（ single-user, co-operatively）、多任务操作系统很类似。这种操作系统上的应用一个失误（misbehaving ），就有可能拖垮整个系统。所以在现代浏览器上一个web页面的失误同样也能做到。拖垮整个浏览器和当前打开的所有tabs只需要一个浏览器或插件的bug.

现代操作系统更健壮，因为它们把应用放在了独立的进程中，它们之间相互隔离。一个应用的崩溃不再会影响其他应用或者整个系统，同时限制每个用户获取其他用户的数据。

Architectural overview 架构预览

我们为浏览器的每一个tab都创建独立进程来避免所有的应用受到渲染引擎bug或故障的干扰。我们同样限制每个渲染引擎访问整个系统中其它的进程。某种意义上，浏览器将会从内存保护和访问控制中受益。

我们称运行UI的主线程、控制tab和插件的进程为“browser进程”或“browser”. 对应的，独立的tab进程称为“渲染进程”或“渲染者（renderer）”. 渲染引擎使用WebKit开源布局引擎来布局HTML.

Managing render processes 管理渲染进程

每一个渲染进程都有一个全局的RenderProcess对象用来管理和父级浏览器进程的通信、维护全局状态。browser 为每一个渲染进程维护一个对应的RenderProcessHost，它用来管理 browser 状态并且和渲染引擎通信。browser和渲染引擎通过Chromium IPC system通信。

Managing views 管理视图

每个渲染引擎都有一个以上的RenderView对象，通过RenderProcess来管理，它对应于tab上的内容。对应的RenderProcessHost为渲染引擎中的每个视图都维护一个对应的RenderViewHost.每个视图都有一个视图ID用来在同一个渲染引擎的不同视图中标示。这个ID在渲染引擎中是唯一的，但在browser中并不是，所以标识一个视图同时需要RenderProcessHost和视图ID. 从browser到特定tab内容的通信通过RenderViesHost对象来完成，它知道如何通过RenderProcesssHost发送信息到RenderView的RenderProcess中。

Components and interfaces 组件和接口

在渲染引擎中：

  • 由RenderProcess处理在 browser 中对应的RenderProcessHostIPC. 每个渲染引擎都有且只有一个RenderProcess对象。这就允许 browser 和渲染引擎双向通信。
  • 由RenderView处理在 browser 中对应的RenderViewHost通信、WebKit内嵌的层。该对象代表一个tab或弹出window中的web页面内容。

在 browser 进程中：

  • Browser对象代表顶层的浏览器window.
  • RenderProcessHost对象代表浏览器方面的单一browser和渲染引擎的IPC连接。browser进程中对应每个渲染引擎都有一个RenderProcessHost.
  • RenderViewHost对象和远端的RenderView封装通信（ encapsulates communication ），RenderWidgetHost为browser 中的RenderWidget处理输入、绘制工作。

访问How Chromium displays web pages设计文档，获取嵌入是如何工作的更多细节。

Sharing the render process 共享渲染进程

通常情况下，每个window或tab都是在新的进程中打开。browser将生成一个新进程并且指示它创建一个RenderView.

有时我们需要在tab或window间共享渲染进程。一个web应用打开新窗口时希望同步通信，例如，在JavaScript中使用window.open. 这种情况下，当我们创建了新的window或tab时，我们需要重用已经打开的window进程。如果已经存在的进程数量过多，我们同样有策略为已经存在的 window 分配新的tab。详见 进程模型.

Detecting crashed or misbehaving renderers 检测渲染引擎的崩溃和错误

每个连接 browser 进程的IPC都检测进程句柄。如果处理了这些句柄，当渲染引擎崩溃的时候，tab将会得到崩溃通知。就目前来说，我们显示的是“sad tab”屏幕，来通知用户渲染引擎崩溃了。可以通过点击重载按钮来重新加载，或者打开一个新的导航（navigation）. 当这种情况发生的时候，我们发现没有进程之后再新建一个。

Sandboxing the renderer 沙盒中的渲染引擎

因为WebKit运行在独立的进程中，我们有机会限制它们访问系统资源。例如我们可以确保渲染引擎只能通过父级的 browser 进程访问网络。同样的，我们可以通过宿主操作系统的内置权限限制它访问文件系统。

进一步限制渲染引擎访问文件系统和网络，我们可以在它访问用户显示器和相关对象的时候设置限制。我们可以在独立的 Windows “Desktop”中运行渲染进程，对用户来说是这透明的。这样就避免了在打开新window或者捕获按键（而做的妥协|泄露渲染引擎）。

Giving back memory 内存回收

运行在独立进程中的渲染引擎，隐藏的tabs会变得较低优先级。通常情况下，在Windows上最小化进程会把它们的内存自动放在”可用内存”池中。在内存不足的情况下，Windows 将会在把它们在高优先级内存中清除之前，和磁盘交换这部分内存，这样有助于保持用户可见的程序响应速度更快。我们同样可以对隐藏的tab使用这样的策略。当渲染进程没有顶级tab的时候，我们可以释放该进程的”working set”大小，提示系统如果需要的话可以交换这部分内存到硬盘。因为我们发现减少工作集大小同样可以减轻tab切换时的性能，当用户在两个tab间切换的时候，我们逐步释放这些内存。这意味着如果用户切换回最近访问过的tab，这个tab的内存比更少访问的tab更有可能paged. 有足够多内存的用户，运行所有应用的时候不会注意到这个过程：Windows只会在需要的时候重新使用这些数据，所以如果有足够多的内存将不会有性能问题。

这帮助我们在低内存情况下获得更优化的内存占用（ memory footprint ）。和不常使用的后台 tab 关联的内存，将在前景（foreground）tab数据可以整体被载入内存的时候整体交换出去。对应的，单个进程的浏览器将会拥有所有tab的数据随机分布在内存中，不可能清晰的分离已用、未用数据，在内存和性能上都有浪费。

Plug-ins 插件

Firefox 风格的NPAPI插件在自己的独立进程中运行，和渲染引擎相分离。详见插件架构

使用hydra简化emacs窗口操作

分割

一直以来在emacs中操作窗口都比较麻烦
垂直分割window：C-x 3
水平分割window：C-x 2

切换

window-numbering

切换window时，如果用的是window-numbering插件，直接使用M-{0-9}就行。这样切换在窗口数比较少的时候还是很方便，但是一旦窗口变多，这种只使用数字的方法容易比较乱。

windmove

所以为了能更直观的切换窗口，可以使用windmove，这是一个库，从emacs 21开始就已经内置

Windmove is a library built into GnuEmacs starting with version 21
emacswiki WindMove

下面的是官方推荐的使用方法，绑定快捷键以后就可以使用，看起来很美，但是如果要快速的移动窗口，C-c的前缀绝对能手抽筋。

ace-window

官方github
这个插件是最全面的，功能最强大，可以删除、交换window，但是各种快捷键也是很麻烦，同时也有使用windmove的问题，就是操作是一次性的，每次的前缀都要求很多。

  • x - delete window
  • m - swap (move) window
  • v - split window vertically
  • b - split window horizontally
  • n - select the previous window
  • i - maximize window (select which window)
  • o - maximize current window

hydra

hydra github
在遇到hydra之前，我一直是ace-window和window-numbering混合使用，虽然能解决一些问题，但是仍然没有顺畅的感觉，直到发现了hydra，所有的操作顿时豁然开朗。

先来看下使用hydra改写以后的键位绑定(以下示例来自youtube视频)：

可以看到几乎所有的window操作：切换、分割、删除、交换、最大化，都已经包含了进来。

调用后的界面为

使用hydra最重要的是从此不再需要不停的敲击前缀C-M-w，前缀敲击一次即可，之后只需要按下双引号里的字母就能执行它对应的命令。

hydra的思路大概类似vim的切换模式，在按下快捷键以后，我们就进入hydra模式，调用的是通过defhydra定义的函数。函数里每一行的格式都为("<key>" command "comment"). 有的会加上 ：color blue，加上这种标注的行为，提示行就会变成蓝色，同时表示点击后会退出hydra.

通过这种方法，我们对window操作时，可以一次C-M-w后，不停的敲击单个字母，来执行以前需要三四个按键才能做的操作，high翻了~

最后是这个函数的使用方式，因为本质上它是把一个函数绑定到了快捷键上，所以还是要放在emacs启动完成以后的hook里面，以避免发生快捷键覆盖。也就是放在我的配置的.emacs.d\init\keybind.el里面。同时记得在最开始(require hydra).

Dijstar算法

计算两点间最短距离
n各节点的图，最笨的方法是遍历n叉树，一共n层，根据每个路径的情况进行判断是否再向下深入遍历。

Dijstar

最原始的方法太笨，耗时、耗空间严重，所以我们尝试使用表来存储
二维表，每一行表示当前节点到其他节点的最短路径
每处理完一个节点，就向下一行

  1. 始化第一行，起始节点I到所有节点的距离为正无穷
  2. 计算第一行，以起点开始，计算和每一个节点的距离，并修改其值，然后把不能直接抵达的节点依旧写正无穷
  3. 选当前行最小值节点作为下一行的起点
  4. 每开始一次都把这个节点排除出去，进入下一行重复查找最小值节点

already-cost(Node-Chosen) + cost(Node-Chosen -> Node-dest) ? already-cost(Node-dest)

解惑

这次有一个节点差一点就是最短路径，可是有一个更近的，选了更近的以后是不是以后的方向都会偏离之前第二短的？是不是以后反而会越来越长？

答：不是，如果以后节点偏离了，计算出来的新值始终没有超过之前差一点就最小的值，那么就不会更新这个值，只有在更优的情况下会更新它

1. 无法合并

Pull is not possible because you have unmerged files.

解决：
git add file.name && git commit -m "removed merge conflicts"

git add完冲突文件以后，才能commit，此时提示符的红色已经消失

2. rebase

之前和一个同事的页面冲突了，以他的为准，所以就把我的文件给删除了。但是我的本地分支比他的提前，git pull以后，还是以我的为准，而我又把文件给删除了，所以此时仓库里是没有那些文件的。

上面的rebase就是我尝试恢复文件的第一个想法，rebase恢复，但是没啥用

2.1 rebase 后的状态

可以看出当前rebase并没有完成，提示如果已经完成的话，就git rebase --continue

有时候也可以git rebase --abort

2.2 rebase 后 git pull 必须指定分支

此时仍然处于rebase，如果要git pull就要指定 git pull origin master

git merge master 之后又提示冲突

3 reset

第二个尝试是 reset：git reset --hard 3b8c58f**514c3fb0022ec5

首先拆分来看，这是两个字符({}+[])[!+[]+!+[]+!+[]]和(![]+[])[!+[]+!+[]+!+[]]
前面一个代表”j”，后面代表”s”
再来拆分:
({}+[])[!+[]+!+[]+!+[]] = ({}+[]) + [!+[]+!+[]+!+[]]
({}+[])输出的是"[object Object]"
[!+[]+!+[]+!+[]]就是把上面的字符串当做数组来取数字，下标为!+[]+!+[]+!+[] = 3
整合起来就是取"[object Object]"下标为3的字符->”j”

同理后面的(![]+[])[!+[]+!+[]+!+[]]就是从"false"中取出下标为3的字符 -> “s”

解释下面的问题

undefined, ``, ‘’ are all false in JavaScript. So now the explanation… As we saw [] + [] is “”, so we might think for [] as for “”. As I mentioned “” is false so !”” is true. The prefix plus converts the boolean value to numer, so +true === 1.

翻译一下：

  • javascript 中 undefined, ``, ‘’ 都是 false
  • [] + [] = “” 即 [] = false
  • !false = true 即 ![] = true
  • + 前缀把布尔值转换为数字，所以 +true === 1
