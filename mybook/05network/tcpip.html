TCP/IP

ACK (Acknowledgement Number)

网络上的传输是没有连接的，包括TCP也是一样的。而TCP所谓的“连接”，其实只不过是在通讯的双方维护一个“连接状态”，让它看上去好像有连接一样。所以，TCP的状态变换是非常重要的

三次链接的握手：保证应用层收到的数据不会因为网络传输而乱序

初始化Sequence Number的初始值，SYN (Synchronize Sequence Numbers)

我的SN是X | 收到你的SN是x+1，我的SN是Y | 收到你的SN是Y+1

四次挥手： 结束 | 收到 | 结束 | 收到

建立连接超时，linux下一共需要63s，TCP才会断开连接

tcp_syncookies应对 SYN FLOOD攻击

RFC793中说，ISN会和一个假的时钟绑在一起，这个时钟会在每4微秒对ISN做加一操作，直到超过2^32，又从0开始。这样，一个ISN的周期大约是4.55个小时

假设我们的TCP Segment在网络上的存活时间不会超过MSL（Maximum Segment Lifetime），所以，只要MSL的值小于4.55小时，那么，我们就不会重用到ISN

TCP重传机制

快重传 Fast Restransmit

SACK Selective Acknowledgment D-SACK Duplicate SACK

RTT算法

Round Trip Time，记录一个包来回的时间，动态改变Timeout -> Retransmission Timeout

Karn / Partridge 算法

死板的忽略重传ACK的时间，只要发生一次重传，就把RTO的时间翻倍

Jacobson / Karels 算法 <- TCP最终采用算法

RTO = 平滑RTT + 4 * 平滑RTT和真实RTT直接的差距

之前两种算法都是“加权”平滑处理，坏处是无法处理大异常

在Linux下，α = 0.125，β = 0.25， μ = 1，∂ = 4 ——参是调整得来的

nobody knows why, it just works

TCP滑动窗口

解决的可靠传输以及包乱序（reordering）的问题

在服务器端的窗口减为0的时候，zero window，就无法再发送

如果server再有窗口，通过ZWP(Zero Window Probe)解决，ZW之后，发送三次window尺寸，三次都为0，TCP就断掉链接

有等待的地方就会有DDoS攻击

Silly Window sydrom

对于以太网来说，MTU是1500字节，除去TCP+IP头的40个字节，真正的数据传输可以有1460，即 MSS（Max Segment Size）

在server的空间很少的时候，每空出几个字节，client就会发送以下，但是数据包的代价也很大，车空人少，浪费带宽。

同时在sender和receiver做处理 Sender：Nagle’s algorithm，延时处理

  1. 要等到 Window Size>=MSS 或是 Data Size >=MSS
  2. 等待时间或是超时200ms

Receiver：直接ack(0)回sender，把window给关闭，等自己攒够了数据，再发送
